Shared Navigation — open a resource once, both libraries follow

component-interop loads Solid Web Components and PodOS and wires their resource channels together — no glue code. Browse on the left; PodOS on the right follows. Open a folder in the Solid Web Components browser and PodOS shows that same folder. This is the broker's resource channel, wired from the two manifests — no capability and no glue. (Sharing the underlying RDF store is a separate pairing — see the Shared Store tab.)

Browse here — Solid Web Components

<sol-pod> file browser. Click a folder to open it.

Follows along — PodOS

<pos-*> shows whatever you open on the left.

<script src="component-interop.js"
        data-stage="local"
        data-manifest="sol-loader.manifest.json pod-os.manifest.json"
        data-bundles="sol-pod @pod-os/elements"
></script>

<!-- swc's browser drives navigation -->
<sol-pod source="https://timbl.solidcommunity.net/public/"></sol-pod>

<!-- PodOS follows the same current resource via the broker's resource channel -->
<pos-app>
  <pos-resource>
    <pos-container-contents></pos-container-contents>
  </pos-resource>
</pos-app>

# solid-web-components manifest — swc ships it; you point data-manifest at it
{
  "interop": {
    "provides": { "resource": { "event": "sol-navigate", "path": "detail.url" } }
  }
}

# pod-os.manifest.json — you ship this small descriptor
{
  "interop": {
    "accepts": { "resource": { "selector": "pos-resource", "attr": "uri" } }
  }
}
